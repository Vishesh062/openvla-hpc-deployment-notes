# 02 — VLA Model Comparison

A summary of the Vision-Language-Action landscape as it stood in early 2025, when the project's model-selection decisions were made. This isn't a reproduction of the papers — read those if you want the math. It's what was weighed against what, and why TinyVLA became the deployment target with OpenVLA as the fine-tune target.

## What a VLA actually is

A VLA model takes an RGB image and a natural-language instruction ("place the red cube on the plate") and returns a sequence of robot actions, usually end-effector deltas and a gripper command. The general shape:

```
image  →  vision encoder  ─┐
                           ├→  multimodal transformer  →  action decoder  →  robot
text   →  language model  ─┘
```

What varies between models: which vision encoder, which language backbone, how actions are decoded (discrete tokens via vector quantisation, or continuous trajectories via diffusion), and what data was used to pretrain.

## The models I looked at

Across weeks 1–5 I read ten papers in detail. The eight that mattered for the deployment decision:

| Model | Vision encoder | Language backbone | Action decoder | Size | Notes |
|---|---|---|---|---|---|
| **OpenVLA** | DINOv2 + SigLIP (`dinosiglip-224px`) | Llama-2-7B | Discrete action tokens (VQ) | 7B | The reference open-source VLA. Trained on OpenX-Embodiment. Heavy but reproducible. |
| **MiniVLA** | SigLIP | Qwen 2.5 0.5B | VQ tokens with **action chunking** | ~1B | Stanford. ~7× smaller than OpenVLA with comparable success rates. Hidden size 896, which is non-standard. |
| **TinyVLA** | CLIP + LLaVA-Pythia-1.3B | Pythia | **Diffusion policy** head | ~1.3B | No robot-specific pretraining required. Strong perf-per-parameter. Modular enough to compose. |
| **CogACT** | SigLIP-based | Llama family | **Decoupled cognitive + action modules** with diffusion | ~7B | Microsoft Research. Smoother action generation from separating cognition and control. |
| **RoboMamba** | Visual tokeniser | Mamba state-space model | Continuous, long-horizon | ~2B | State-space architecture, not transformer. Efficient long-context reasoning. |
| **VLMimic** | CLIP | LLM-based | Imitation from human video | — | Specialises in learning from in-the-wild human video demos, not robot teleop. |
| **Dreamitate** | — | — | **Diffusion video generation** as policy | — | Generates imagined future video frames, extracts policy from them. |
| **RoboCLIP** | CLIP | — | Contrastive language-image grounding | — | A building block, not an end-to-end policy. |

Two further papers were context but not direct candidates: "Show and Tell" (2024), more about modular policy synthesis from video + verbal instruction, and Multi-View Masked World Models, which is world-modelling for robotic learning.

## What actually mattered for selection

Five things drove the choice:

1. **Deployable on dual A100s.** 7B models fit at FP16 (~14 GB). Training one from scratch was out of scope. For the fine-tune target I wanted something under 2B.
2. **Open code and weights.** No room to reimplement a paper.
3. **Compatibility with ALOHA-style demonstrations.** The dataset format (image + instruction + 7-dim end-effector vector) had to match what the model expected.
4. **Inference latency.** Real-time was the eventual goal; under 1 second per inference was the prototyping target.
5. **Extensibility.** The hybrid pipeline plan (see `03-hybrid-pipeline-design.md`) needed a model whose perception module could feed a separate planning module cleanly.

## The architectural choice that matters most: how actions are decoded

This is the dimension that most affects deployment behaviour. Worth getting clear on before the rest of the project makes sense.

**Discrete token decoding (OpenVLA, MiniVLA).** Continuous end-effector deltas get quantised into a finite vocabulary of action tokens. The model generates these autoregressively, like text. Pros: reuses the language model's existing token machinery; training is stable. Cons: quantisation noise; smoothness depends on bin granularity. MiniVLA improves on this with **action chunking** — predicting multiple action tokens at once rather than one-by-one. Cuts inference steps significantly.

**Diffusion policy decoding (TinyVLA, CogACT).** Actions get generated by iterative denoising of a noise vector conditioned on the visual-language context. Pros: smoother trajectories, less quantisation artifact. Cons: multi-step sampling adds latency unless distilled; harder to train stably.

For ALOHA, where smoothness matters, diffusion-based generation has obvious appeal. But the multi-step sampling cost is real, which is why CogACT's distillation work and TinyVLA's lightweight design were both interesting.

## Why TinyVLA for the local prototype

TinyVLA hit the deployment criteria best:

- Compact (~1.3B vs OpenVLA's 7B), so viable on more modest hardware than the eventual fine-tune target
- No robot pretraining required, so we could deploy and test without sourcing OpenX-Embodiment-scale data
- Diffusion policy, so smoother actions for teleop data
- Modular architecture, so perception could be cleanly separated from action generation (the hybrid pipeline in `docs/03` depends on this)
- Runnable codebase, not just describable

The local backend work in weeks 7–8 was built around TinyVLA. The image → action token loop running through a FastAPI server and published over ROS 2 topics was the first thing in the project that did anything end-to-end.

## Why OpenVLA for the fine-tune target

Once the local prototype was working, the centre's interest shifted to fine-tuning on `aloha_clean_dish` for comparison against MiniVLA. OpenVLA was the right choice because:

- It's the reference implementation. Every downstream VLA (including MiniVLA) builds on its training scaffolding.
- The codebase is well-maintained with documented fine-tuning entry points (`vla-scripts/finetune.py`).
- A 7B baseline on a custom dataset is a meaningful signal even if the absolute numbers don't beat published benchmarks.
- The plan was always to use OpenVLA's training infrastructure to fine-tune the smaller MiniVLA. That turned out to be more painful than expected. See `docs/06`.

## Caveats

Three things worth saying plainly:

1. The benchmark numbers in each paper aren't directly comparable — different evaluation suites, different physical platforms, different demonstration densities. The ranking above reflects deployment fit for ALOHA, not absolute capability.
2. The field moved fast during the project. Variants and successors of these models were published mid-internship. This is a snapshot, not a survey.
3. I read these papers as a Master's student, not a domain expert. The architectural summaries are informed but not authoritative. Original papers remain the source of truth.
